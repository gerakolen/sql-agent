# Политика безопасности

## Обзор

HomoSQL Agent реализует многоуровневую систему безопасности для обеспечения безопасного взаимодействия с SQL-базами данных.

## Меры безопасности

### 1. Защита от SQL-инъекций

#### Валидация SQL
Класс [`SQLValidator`](src/validators/sql_validator.py:10) предоставляет комплексную валидацию SQL-запросов:

- **Запрещённые ключевые слова**: Блокирует операции модификации данных (DROP, DELETE, UPDATE, INSERT, ALTER, CREATE, TRUNCATE, GRANT, REVOKE, EXEC, EXECUTE)
- **Только SELECT**: Все запросы должны начинаться с SELECT
- **Обнаружение подозрительных паттернов**: Блокирует SQL-комментарии, множественные запросы и расширенные хранимые процедуры
- **Автоматический LIMIT**: Добавляет LIMIT для предотвращения избыточного извлечения данных

```python
FORBIDDEN_KEYWORDS = [
    "DROP", "DELETE", "UPDATE", "INSERT", "ALTER", "CREATE", 
    "TRUNCATE", "GRANT", "REVOKE", "EXEC", "EXECUTE"
]
```

#### Параметризованные запросы
Класс [`DatabaseConnection`](src/database/connection.py:14) использует параметризованные запросы SQLAlchemy:

```python
result = connection.execute(text(query), params or {})
```

Это предотвращает SQL-инъекции за счёт правильного экранирования пользовательского ввода.

### 2. Контроль доступа

#### Доступ только для чтения
- Агент разработан для операций только чтения
- Системные промпты явно запрещают DML-операции
- Валидация SQL блокирует операции модификации

#### Ограничение результатов
- Максимальное количество результатов по умолчанию: 100 строк (настраивается через переменную окружения `MAX_RESULTS`)
- Принудительное добавление LIMIT

### 3. Безопасность конфигурации

#### Переменные окружения
Чувствительная конфигурация хранится в переменных окружения (файл `.env`):

- Учётные данные базы данных
- API-ключи
- Base URL

Файл `.env` исключён из системы контроля версий через [`.gitignore`](.gitignore:37).

#### Валидация конфигурации
Класс [`LLMConfig`](src/config.py:38) проверяет наличие обязательных API-ключей:

```python
def __post_init__(self):
    if self.provider == "openai" and not self.api_key:
        raise ValueError("OPENAI_API_KEY не установлен в переменных окружения")
```

### 4. Безопасность логирования

#### Уровень логирования по умолчанию
- Уровень логирования по умолчанию: WARNING (логи INFO и DEBUG скрыты)
- Логи отображаются только с флагом `--logs`
- Предотвращает утечку чувствительной информации при обычной работе

#### Отсутствие чувствительных данных в логах
- Пароли базы данных никогда не логируются
- API-ключи никогда не логируются
- Результаты запросов не логируются по умолчанию

### 5. Обработка ошибок

#### Надёжная обработка ошибок
- Ошибки подключения к базе данных перехватываются и логируются
- Ошибки валидации SQL возвращают понятные сообщения
- Стек-трейсы не раскрываются конечным пользователям

#### Управление подключениями
- Пул соединений с `pool_pre_ping=True`
- Контекстные менеджеры для управления сессиями
- Автоматическая очистка подключений

## Рекомендации по безопасности

### Для пользователей

1. **Никогда не коммитить файл `.env`** - Используйте `.env.example` как шаблон
2. **Используйте пользователей базы данных только для чтения** - Создайте отдельных пользователей базы данных с правами только SELECT
3. **Установите подходящее значение `MAX_RESULTS`** - Ограничьте количество возвращаемых данных
4. **Проверяйте сгенерированный SQL** - Используйте флаг `--sql` для проверки запросов перед выполнением
5. **Обновляйте зависимости** - Регулярно обновляйте `requirements.txt`

### Для разработчиков

1. **Валидация SQL** - Всегда валидируйте SQL-запросы перед выполнением
2. **Параметризованные запросы** - Никогда не объединяйте пользовательский ввод в строки SQL
3. **Переменные окружения** - Храните все чувствительные данные в переменных окружения
4. **Логирование** - Избегайте логирования чувствительной информации
5. **Валидация ввода** - Проверяйте все пользовательские данные

## Известные ограничения

1. **Запросы, генерируемые LLM**: ИИ генерирует SQL-запросы, которые могут содержать ошибки. Всегда проверяйте запросы в продакшене.
2. **Сложные запросы**: Очень сложные запросы могут превысить правила валидации.
3. **Специфичные функции базы данных**: Некоторые специфичные функции базы данных могут быть не полностью проверены.

## Сообщение о проблемах безопасности

Если вы обнаружили уязвимость безопасности, пожалуйста:

1. Не создавайте публичную issue
2. Отправьте детали владельцам проекта приватно
3. Включите шаги для воспроизведения уязвимости
4. Дайте время на устранение проблемы до публичного раскрытия
